# ADR-0001: Hybrid Python + C++ Architecture

Дата: 2026-07-07
Статус: принято

## Контекст

Проект `trading-robot-lab` создается как исследовательская и техническая база для разработки торгового робота.

Возник ключевой архитектурный вопрос:

```text
Делать систему полностью на Python, полностью на C++, или использовать гибридный подход?
```

Python удобен для исследований и бэктеста, но слишком медленный и нестабильный по задержкам для настоящего low-latency/HFT execution.

C++ быстрый и лучше подходит для будущего боевого контура, но писать весь проект только на C++ с самого начала неэффективно: исследование стратегий, обработка данных, графики, отчеты и подбор параметров будут развиваться слишком медленно.

## Решение

Принять гибридную архитектуру:

```text
Python = research/backtest/analysis/prototyping
C++    = future low-latency core/risk/execution/replay
```

Python-часть остается в проекте и используется как лаборатория для быстрой проверки гипотез.

C++-часть будет добавлена отдельным модулем `cpp/` и сначала будет работать только в безопасном paper/backtest/demo-режиме, без реального брокера и без live-заявок.

## Базовое разделение модулей

Целевая схема:

```text
Market Data Gateway
        -> Strategy Engine
        -> Risk Engine
        -> Execution Gateway
        -> Portfolio / PnL
        -> Logs / Storage / Monitoring
```

Принципиальные правила:

1. Стратегия не отправляет заявки напрямую.
2. Risk Engine является обязательным отдельным фильтром.
3. Execution Gateway является единственным модулем, который может отправлять заявки.
4. Live-режим выключен по умолчанию.
5. Backtest, paper и live должны использовать максимально одинаковую стратегическую логику.

## Последствия

### Положительные

- Можно быстро проверять торговые идеи в Python.
- Можно постепенно строить быстрый C++ core.
- Архитектура не привязана к одному источнику данных.
- Стратегии можно тестировать на исторических и реальных данных через единый интерфейс.
- Risk Engine отделен от стратегии и исполнения.
- Проект не превращается в хаотичный монолит.

### Отрицательные

- Нужно поддерживать согласованность типов между Python и C++.
- Появляется две технологические зоны.
- Нужно дисциплинированно документировать контракты данных.
- Нельзя слепо переносить стратегию из Python в C++ без тестов на эквивалентность.

## Что делать следующим шагом

Создать C++ skeleton:

```text
cpp/CMakeLists.txt
cpp/include/trl/types.hpp
cpp/include/trl/risk_engine.hpp
cpp/include/trl/synthetic_spread_strategy.hpp
cpp/include/trl/paper_execution_gateway.hpp
cpp/src/risk_engine.cpp
cpp/src/synthetic_spread_strategy.cpp
cpp/src/paper_execution_gateway.cpp
cpp/apps/paper_demo.cpp
```

Первый C++ demo-контур должен быть безопасным:

```text
MarketEvent -> Strategy -> Risk -> PaperExecution -> Portfolio
```

Он не должен подключаться к брокеру и не должен отправлять реальные заявки.

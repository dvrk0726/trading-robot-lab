## Главная цель проекта

Создать новую архитектуру торгового робота для исследования стратегий, бэктестинга, симуляции, риск-контроля и только затем осторожного подключения к реальному рынку.

Проект не должен восприниматься как готовый робот для запуска на реальном торговом счете.

## Ключевой принцип разработки

Сначала исследование и безопасность. Потом тестирование. Только потом торговля.

Порядок этапов:

1. Документация архитектуры.
2. Анализ старого HFT-робота.
3. Выделение торговой идеи отдельно от старого кода.
4. Проектирование нового ядра.
5. Бэктест на исторических данных.
6. Replay-симуляция с учетом стакана, задержек, комиссий и проскальзывания.
7. Paper trading / sandbox.
8. Live-режим только после отдельного ручного подтверждения владельца проекта.

## Что известно про старый архивный HFT-робот

В проекте есть старый архивный робот под российский срочный рынок, предположительно ориентированный на фьючерс RI и индексный/статистический арбитраж.

Старый проект содержит:

- C# / WinForms приложение `new_robot_uralpro`.
- Логику получения данных через QUIK DDE, TRANS2QUIK и Plaza2/ClientGate.
- Логику отправки заявок.
- Расчет синтетического индекса.
- Расчет спреда между фьючерсом и синтетическим индексом.
- Выставление и перестановку лимитных заявок.
- Учет позиции.
- Закрытие позиции перед клирингом/концом торгового периода.
- Старые логи и CSV с результатами.

Старый робот НЕ считается готовым к использованию. Его роль в проекте:

- источник исторической архитектуры;
- источник торговой идеи;
- материал для анализа;
- пример того, как была устроена старая инфраструктура робота.

Старый робот не должен запускаться на реальном торговом счете.

## Главная торговая идея из старого проекта

Базовая идея старого робота:

1. Рассчитать синтетический индекс на основе корзины акций и валютного фактора.
2. Сравнить цену фьючерса RI с расчетным синтетическим индексом.
3. Получить спред.
4. Если спред сильно отклоняется от нормального значения, искать вход лимитной заявкой.
5. При нормализации спреда или по риск-условиям выходить из позиции.

Условно:

```text
spread = futures_price - coefficient * synthetic_index
```

Если `spread` слишком высокий — возможен сигнал на продажу фьючерса.
Если `spread` слишком низкий — возможен сигнал на покупку фьючерса.

Эта идея должна быть заново проверена на современных данных. Нельзя считать, что она работает сейчас.

## Почему нельзя просто запускать старого робота

Причины:

- устаревшая инфраструктура: QUIK DDE, TRANS2QUIK, старый Plaza2/ClientGate;
- рынок изменился;
- состав индекса изменился;
- комиссии и ликвидность изменились;
- современные конкуренты быстрее;
- нет полноценного современного risk engine;
- нет нормального replay-бэктеста;
- нет подтвержденной актуальной прибыльности;
- есть риск отправки реальных заявок при неправильном подключении.

## Требования к новой архитектуре

Новая система должна быть модульной.

Минимальные модули:

```text
market_data/      получение и нормализация рыночных данных
strategy/         расчет сигналов
risk/             риск-менеджмент и лимиты
execution/        заявки, отмены, модификации
portfolio/        позиции, сделки, PnL
backtest/         историческое тестирование
replay/           симуляция рынка и исполнения
storage/          хранение данных, сделок, результатов
monitoring/       логи, метрики, алерты
config/           настройки без секретов
```

Код стратегии не должен напрямую отправлять заявки на биржу. Стратегия должна только выдавать намерение/сигнал. Исполнение должно проходить через risk engine и execution gateway.

## Обязательные режимы работы

Система должна поддерживать минимум три режима:

```text
backtest     исторический тест без реального рынка
paper        симуляция на текущих/псевдореальных данных без реальных заявок
live         реальная торговля, выключена по умолчанию
```

Live-режим должен быть максимально защищен и не должен включаться случайно.

## Базовые правила безопасности

Запрещено хранить в репозитории:

- `.env` с реальными секретами;
- API-ключи;
- брокерские токены;
- логины и пароли;
- приватные ключи;
- доступы к VPS;
- реальные ключи подписи;
- персональные данные клиентов;
- данные реального брокерского счета без необходимости и обезличивания.

В репозитории можно хранить только пример:

```text
.env.example
```

Реальный `.env` должен быть добавлен в `.gitignore`.

## Минимальный risk management для будущего робота

До live-режима должны быть реализованы:

- максимальный дневной убыток;
- максимальная позиция;
- максимальное количество заявок в минуту;
- запрет торговли при отсутствии свежих рыночных данных;
- запрет торговли при рассинхроне данных;
- kill switch;
- ручная кнопка остановки;
- логирование всех решений стратегии;
- логирование всех заявок;
- логирование всех ответов биржи/брокера;
- отдельный флаг включения live-режима;
- защита от повторной отправки заявки;
- проверка аномальных цен.

## Подход к разработке

Правильный порядок работы:

1. Сначала описать архитектуру.
2. Затем описать стратегию словами и формулами.
3. Затем сделать прототип расчета сигналов.
4. Затем сделать бэктест.
5. Затем проверить на данных.
6. Затем добавить risk engine.
7. Затем добавить paper trading.
8. Затем только обсуждать подключение к реальному рынку.

Нельзя начинать с подключения к брокеру.
Нельзя начинать с реальных заявок.
Нельзя переносить старый код без анализа.

## Важное предупреждение

Этот проект связан с алгоритмической торговлей и потенциальным финансовым риском.

Любой код, связанный с реальными заявками, должен рассматриваться как высокорисковый. Перед использованием на реальном счете требуется отдельная проверка, тестирование, лимиты риска и ручное подтверждение владельца проекта.

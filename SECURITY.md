# SECURITY

Дата создания: 2026-07-07
Репозиторий: `dvrk0726/trading-robot-lab`
Статус: обязательные правила безопасности проекта

## Назначение файла

Этот файл описывает правила безопасности для разработки торгового робота.

Проект связан с алгоритмической торговлей, потенциальными брокерскими доступами, рыночными данными, торговыми логами и будущим исполнением заявок. Поэтому любые секреты, доступы и исполняемые файлы должны обрабатываться строго.

## Главный принцип

В GitHub храним только код, документацию, безопасные примеры конфигурации и исследовательские материалы.

В GitHub не храним реальные секреты, ключи, доступы, токены, пароли и файлы, которые могут случайно запустить торговую или вредоносную логику.

## Что запрещено коммитить

Нельзя добавлять в репозиторий:

```text
.env
.env.local
.env.production
.env.live
*.key
*.pem
*.pfx
*.p12
*.crt
*.cer
id_rsa
id_ed25519
```

Нельзя добавлять:

- API-ключи;
- брокерские токены;
- логины и пароли;
- пароли от VPS;
- SSH-ключи;
- приватные сертификаты;
- ключи электронной подписи;
- access token / refresh token;
- webhook secrets;
- реальные cookies;
- реальные session-файлы;
- конфиги с live-доступами;
- реальные реквизиты брокерского счета;
- персональные данные клиентов;
- сканы документов;
- финансовые документы с персональными данными без обезличивания.

## Исполняемые и бинарные файлы

В репозиторий запрещено добавлять без отдельного решения:

```text
*.exe
*.dll
*.msi
*.bat
*.cmd
*.ps1
*.so
*.dylib
*.jar
*.bin
```

Причина: старые торговые архивы могут содержать исполняемые файлы, которые способны отправлять реальные заявки, загружать неизвестные библиотеки или выполнять нежелательные действия.

Если бинарный файл нужен для анализа, его нельзя сразу класть в основной репозиторий. Сначала нужно:

1. Проверить файл в изолированной среде.
2. Зафиксировать происхождение файла.
3. Описать назначение.
4. Получить отдельное ручное подтверждение владельца проекта.
5. Добавить файл только в специальную исследовательскую зону, если это действительно необходимо.

## Старый HFT-архив

Старый архивный HFT-робот нельзя загружать в репозиторий целиком без фильтрации.

Особенно опасны:

- `.exe`;
- `.dll`;
- старые конфиги;
- файлы с путями к рабочим терминалам;
- файлы с брокерскими настройками;
- реальные торговые логи;
- любые файлы с идентификаторами счетов.

Правильный порядок работы со старым архивом:

1. Сначала анализировать локально или в изолированной среде.
2. Вытащить только безопасный исходный код и документацию.
3. Удалить бинарники и секреты.
4. Добавить в репозиторий только очищенную версию.
5. Описать перенос в `research/old_robot_analysis/README.md`.

## Файлы конфигурации

В репозитории разрешен только пример конфигурации:

```text
.env.example
```

Пример должен содержать пустые или фейковые значения:

```env
TRADING_MODE=backtest
BROKER_API_KEY=put_key_here
BROKER_ACCOUNT_ID=put_account_here
MAX_DAILY_LOSS=1000
LIVE_TRADING_ENABLED=false
```

Реальный `.env` должен храниться только локально или на сервере и быть исключен через `.gitignore`.

## Live-режим

Любой live-режим должен быть выключен по умолчанию.

Безопасное значение по умолчанию:

```env
TRADING_MODE=backtest
LIVE_TRADING_ENABLED=false
```

Запрещено делать так, чтобы робот после клонирования репозитория мог сразу отправлять реальные заявки.

Для включения live-режима в будущем должны требоваться минимум два явных условия:

```env
TRADING_MODE=live
LIVE_TRADING_ENABLED=true
```

Дополнительно желательно требовать ручное подтверждение в интерфейсе или командной строке.

## Risk-first правило

До появления live-режима должны существовать:

- максимальный дневной убыток;
- максимальная позиция;
- максимальный размер одной заявки;
- лимит количества заявок в минуту;
- kill switch;
- проверка свежести рыночных данных;
- проверка аномальных цен;
- запрет торговли при потере связи;
- логирование всех решений стратегии;
- логирование всех заявок;
- логирование всех ответов брокера/биржи.

Без этих элементов live-режим считается запрещенным.

## Проверка перед коммитом

Перед каждым коммитом нужно проверить, что в изменения не попали секреты.

Минимальная ручная проверка:

```bash
git status
git diff --staged
```

Нужно искать строки вида:

```text
API_KEY=
SECRET=
TOKEN=
PASSWORD=
PASS=
LOGIN=
PRIVATE_KEY=
BROKER=
ACCOUNT_ID=
ACCESS_TOKEN=
REFRESH_TOKEN=
```

Если такие строки появились, нужно убедиться, что значения фейковые и безопасные.

## Работа с рыночными данными

Исторические данные, тиковые данные, стаканы и торговые логи могут быть большими и потенциально чувствительными.

По умолчанию их не нужно хранить в GitHub.

Допустимо хранить:

- маленькие синтетические примеры;
- обезличенные sample-файлы;
- схемы таблиц;
- описание форматов;
- инструкции по загрузке данных.

Недопустимо без отдельного решения хранить:

- реальные сырые тиковые данные большого объема;
- реальные брокерские отчеты;
- полные журналы реальных сделок;
- данные с идентификаторами счета.

## Работа с внешними ИИ-агентами

Если к проекту подключается другой ИИ-агент, он должен:

1. Прочитать `AI_CONTEXT.md`.
2. Прочитать `PROJECT_STATE.md`.
3. Прочитать этот файл.
4. Не добавлять секреты.
5. Не включать live-режим.
6. Не предлагать запуск на реальном счете без risk engine и бэктеста.
7. Не переносить старые `.exe` и `.dll` в основной репозиторий.
8. Любые рискованные действия выносить на ручное подтверждение владельца проекта.

## Что делать при случайной утечке секрета

Если в репозиторий случайно попал ключ, токен, пароль или доступ:

1. Немедленно считать секрет скомпрометированным.
2. Отозвать или заменить ключ у поставщика/брокера/сервера.
3. Удалить секрет из текущей версии файлов.
4. Не считать обычное удаление коммита достаточной защитой.
5. Проверить историю репозитория.
6. При необходимости очистить историю Git.

Важно: если секрет уже был загружен в GitHub, его нужно менять. Просто удалить строку из нового коммита недостаточно.

## Текущий статус безопасности

- репозиторий приватный;

- брокерские доступы не добавлены;
- реальные секреты не должны храниться в репозитории;
- следующий обязательный шаг — создать `.gitignore` и `.env.example`.

# Hybrid Architecture: Python Research + C++ Execution Core

Дата создания: 2026-07-07
Статус: базовое архитектурное направление проекта

## Назначение документа

Этот документ фиксирует выбранный подход к архитектуре торгового робота, чтобы новый ИИ-агент или разработчик сразу понял, почему проект не должен быть полностью на Python и почему не нужно сразу писать всё на C++.

Короткий вывод:

```text
Python = исследование, бэктест, анализ данных, отчеты, быстрые прототипы.
C++    = будущий быстрый контур: market data -> strategy -> risk -> execution.
```

Проект строится как гибридная торговая лаборатория, а не как один монолитный робот.

## Почему не только Python

Python удобен для исследования, но не подходит как единственный язык для низколатентной боевой торговли.

Ограничения Python для HFT/low-latency:

- GIL;
- сборщик мусора;
- нестабильные задержки;
- высокая стоимость абстракций;
- зависимость от библиотек;
- сложность гарантировать микросекундную реакцию;
- риск случайных пауз в критический момент.

Python остается полезен для:

- исследования торговых гипотез;
- расчета статистики;
- загрузки и подготовки исторических данных;
- бэктеста;
- визуализации;
- подбора параметров;
- отчетов;
- быстрой проверки идей.

Вывод: Python в проекте — это лаборатория, а не финальный HFT-двигатель.

## Почему не только C++

C++ быстрее и лучше подходит для execution/core, но начинать весь проект только с C++ нерационально.

Минусы C++ на раннем исследовательском этапе:

- дольше писать;
- дольше менять гипотезы;
- сложнее анализировать данные;
- сложнее делать графики и отчеты;
- выше риск технических ошибок;
- каждая проверка идеи занимает больше времени.

Главный риск на старте — не скорость кода, а отсутствие рабочей торговой идеи.

Если стратегия не имеет преимущества после комиссий, проскальзывания и задержек, быстрый C++ не спасет. Он просто будет быстро исполнять плохую стратегию.

Вывод: C++ нужен, но после того, как архитектура и гипотезы будут проверены.

## Базовая профессиональная схема

Система должна быть разделена на независимые модули.

```text
Historical Data / Real Market Data
              |
              v
Market Data Gateway
              |
              v
Strategy Engine
              |
              v
Risk Engine
              |
              v
Execution Gateway
              |
              v
Portfolio / PnL
              |
              v
Logs / Storage / Monitoring
```

Главное правило:

```text
Стратегия не отправляет заявки напрямую.
```

Стратегия только формирует сигнал или намерение. Заявка проходит через risk engine, а затем через execution gateway.

## Модуль 1. Market Data Gateway

Назначение: получать, нормализовать и отдавать рыночные события.

Источник данных может быть разный:

- реальные рыночные данные;
- исторические данные;
- replay-режим;
- синтетические demo-данные;
- эмулятор.

Стратегия не должна знать, откуда пришли данные. Она получает единый тип события:

```text
MarketEvent / MarketSnapshot
```

Примеры данных:

- последняя цена;
- bid/ask;
- стакан;
- сделки;
- фьючерс;
- синтетический индекс;
- акции из корзины;
- валютный фактор.

## Модуль 2. Strategy Engine

Назначение: принимать рыночные события и выдавать торговое намерение.

Стратегия может выдавать:

```text
HOLD
BUY
SELL
EXIT
```

Или более строго:

```text
OrderIntent
```

Пример:

```text
Хочу купить 1 контракт по цене X, потому что spread ушел ниже нормального диапазона.
```

Стратегия не должна:

- обращаться к брокеру;
- отправлять реальные заявки;
- менять позицию напрямую;
- обходить risk engine.

## Модуль 3. Risk Engine

Назначение: фильтровать любые торговые намерения до отправки заявки.

Risk Engine должен быть отдельным обязательным модулем.

Минимальные проверки:

- максимальная позиция;
- максимальный размер заявки;
- максимальный дневной убыток;
- количество заявок в минуту;
- свежесть рыночных данных;
- аномальная цена;
- рассинхрон данных;
- запрет live-режима без явного включения;
- защита от дублирующих заявок;
- kill switch.

Даже если стратегия ошиблась, risk engine должен блокировать опасное действие.

## Модуль 4. Execution Gateway

Назначение: единственная часть системы, которая имеет право отправлять заявки.

Execution Gateway должен уметь:

- выставить заявку;
- отменить заявку;
- изменить/переставить заявку;
- получить статус заявки;
- получить сделку/fill;
- синхронизировать позицию;
- вести журнал отправленных команд.

Execution Gateway должен иметь режимы:

```text
backtest
paper
live
```

Live-реализация должна быть отдельной, выключенной по умолчанию и защищенной дополнительными условиями.

## Модуль 5. Backtest / Replay Engine

Назначение: проверять стратегию на исторических данных.

Важный принцип:

```text
Одна и та же стратегия должна использоваться в backtest, paper и live.
```

Иначе тесты будут проверять одно, а реальная торговля будет делать другое.

Развитие backtest/replay должно идти по уровням:

### Уровень 1. Простой бэктест

- данные идут по времени;
- сделки считаются по цене снимка;
- стакан не моделируется;
- подходит только для проверки архитектуры.

### Уровень 2. Бэктест с комиссиями и проскальзыванием

- добавляются комиссии;
- добавляется простой slippage;
- добавляется учет позиции и PnL.

### Уровень 3. Replay со стаканом

- исторический order book;
- очередь заявок;
- частичные исполнения;
- отмены;
- задержки;
- реалистичное исполнение.

### Уровень 4. Latency simulation

- задержка данных;
- задержка стратегии;
- задержка отправки заявки;
- конкуренция за очередь;
- проверка чувствительности стратегии к скорости.

## Модуль 6. Storage / Logs / Monitoring

Назначение: хранить все важные события и результаты.

Что нужно логировать:

- входящие market events;
- сигналы стратегии;
- решения risk engine;
- заявки;
- отмены;
- fills;
- позицию;
- PnL;
- ошибки;
- задержки;
- параметры стратегии.

Возможные хранилища:

```text
Parquet      исторические данные и исследовательские датасеты
PostgreSQL   заявки, сделки, позиции, настройки, результаты
ClickHouse   большие объемы тиковых данных, если потребуется
CSV          только маленькие sample-файлы, не реальные приватные данные
```

На старте достаточно файлов и простого формата. Базу данных нужно подключать после появления реальных объемов данных и понятной схемы.

## Разделение языков

### Python-часть

Python используется для:

- исследования;
- быстрых прототипов;
- бэктеста первого уровня;
- загрузки и анализа исторических данных;
- графиков;
- отчетов;
- сравнения параметров;
- работы другого ИИ с логикой стратегии.

Текущий Python-каркас уже создан в:

```text
src/trading_robot_lab/
examples/
tests/
```

### C++-часть

C++ будет использоваться для будущего быстрого ядра:

- обработка market events;
- быстрая стратегия;
- risk engine;
- paper execution;
- replay execution;
- позже — live execution gateway.

## Что делать сейчас

Текущий правильный путь:

1. Не удалять Python-часть.
2. Использовать Python как лабораторию.
3. Добавить C++ skeleton без брокера и без live-торговли.
4. Сделать одинаковые базовые типы в Python и C++:

```text
MarketEvent
StrategySignal
OrderIntent
RiskDecision
ExecutionReport
Fill
PortfolioState
```

5. Сначала реализовать C++ paper/demo-контур:

```text
MarketEvent -> Strategy -> Risk -> PaperExecution -> Portfolio
```

6. Только после этого обсуждать реальные данные и live gateway.

## Что не делать сейчас

Не нужно сейчас:

- подключать брокера;
- подключать реальные заявки;
- переносить старый HFT-робот целиком;
- загружать `.exe` и `.dll` из старого архива;
- оптимизировать микросекунды без доказанной стратегии;
- строить сложную базу данных до появления формата данных;
- делать live-режим.

## Целевая идея проекта

Проект должен развиваться как торговая исследовательская платформа:

```text
Research -> Backtest -> Replay -> Paper -> Live Gate
```

Live Gate — это отдельный рубеж, который нельзя проходить автоматически.

## Архитектурный вывод

Оптимальный подход:

```text
Гибридная архитектура.
Python остается для исследования и бэктеста.
C++ добавляется как будущий быстрый контур исполнения.
Risk Engine является отдельным обязательным модулем.
Execution Gateway является единственным модулем, который может отправлять заявки.
Backtest и live должны использовать максимально одинаковую стратегическую логику.
```
